# Code Review: Buggy Implementation

## The Buggy Code

```go
func (uc *Interactor) Execute(ctx context.Context, req *Request) error {
    order, _ := uc.repo.Retrieve(ctx, req.OrderID)

    order.status = StatusCompleted
    order.completedAt = time.Now()

    if err := uc.db.Update(order); err != nil {
        return err
    }

    event := OrderCompletedEvent{
        OrderID:    order.id,
        CustomerID: order.customerID,
        Total:      order.totalAmount,
    }

    if err := uc.eventBus.Publish(event); err != nil {
        return err 
    }

    return nil
}
````

## 1. Why Direct Publish Breaks Reliability

Обновление базы данных и публикация события — это две независимые операции без общей транзакционной гарантии. Любая из них может выполниться успешно, а другая — нет. Это создаёт окно, в котором сохранённое состояние системы и доставка событий расходятся — и это состояние может никогда не восстановиться автоматически.

Дополнительные проблемы в данном коде:

* Ошибка `Retrieve` полностью игнорируется (`order, _ :=`), что может привести к panic при работе с `nil`.
* Изменение состояния происходит напрямую, минуя доменный метод агрегата — бизнес-правила не проверяются.
* Используется `time.Now()`, что делает поведение недетерминированным и плохо тестируемым.
* Событие создаётся вне домена, что разрывает связь между бизнес-фактом и агрегатом, которому он принадлежит.

## 2. The Exact Failure Scenario

1. `uc.db.Update(order)` выполняется успешно — заказ теперь в состоянии `StatusCompleted` в базе.
2. `uc.eventBus.Publish(event)` падает — таймаут, недоступность брокера, падение процесса и т.д.
3. Функция возвращает ошибку вызывающему коду.

**Результат:** Заказ навсегда помечен как завершённый в базе, но `OrderCompletedEvent` никогда не был доставлен. Downstream-системы (биллинг, доставка, уведомления) не узнают о завершении. Механизма обнаружения или восстановления этой рассинхронизации нет, потому что сам факт намерения отправить событие не был зафиксирован.

Обратная ситуация также возможна: событие успешно публикуется, но обновление базы не фиксируется (или процесс падает до commit). В результате downstream получает событие о завершении заказа, которое фактически не произошло.

## 3. Why Outbox Must Be in the Same Transaction

Паттерн outbox решает эту проблему, фиксируя намерение отправить событие в виде записи в таблице outbox, которая создаётся в **той же транзакции**, что и обновление агрегата.

* Если транзакция коммитится, и обновление заказа, и запись в outbox становятся долговечными. Фоновый воркер позже прочитает запись и отправит событие.
* Если транзакция откатывается, ни обновление заказа, ни запись в outbox не сохраняются. «Фантомное» событие не может появиться.

Это устраняет окно между «состояние изменено» и «событие опубликовано». Запись в outbox — это долговечный факт намерения, который затем преобразуется воркером в доставку (с at-least-once семантикой).

Если бы вставка в outbox выполнялась в **отдельной транзакции**, проблема рассинхронизации вернулась бы: одна операция могла бы закоммититься без другой.
