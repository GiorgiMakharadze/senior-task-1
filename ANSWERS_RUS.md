# Answers

## Q1: At-Least-Once vs Exactly-Once

Outbox worker выполняет два шага для каждой записи:

1. **Публикует** событие в брокер сообщений.
2. **Помечает как обработанное** в таблице outbox.

Если воркер падает после шага 1, но до шага 2, запись остаётся необработанной. После рестарта воркер снова её прочитает и повторно опубликует. Консьюмер получит дубликат.

Это модель **at-least-once** доставки: каждое событие гарантированно будет доставлено как минимум один раз, но возможны дубликаты. Это не exactly-once, потому что нет атомарной операции, охватывающей одновременно «публикацию в брокер» и «обновление статуса в базе».

**Как с этим работать на стороне консьюмера:**

- **Идемпотентная обработка:** Консьюмер должен корректно обрабатывать повторные события. Повторное применение `OrderCompletedEvent` не должно приводить к двойному списанию или дублированию уведомлений.
- **Дедупликация по event ID:** Хранить обработанные event ID (например, в таблице `processed_events`). Перед выполнением сайд-эффектов проверять, был ли уже обработан этот ID.
- **Идемпотентные ключи для внешних вызовов:** При вызове внешних систем (платёж, email) использовать event ID или производный ключ, чтобы внешняя система тоже отбрасывала дубликаты.

## Q2: Outbox Table Schema

```sql
CREATE TABLE outbox (
    id             STRING(36)    NOT NULL,
    aggregate_type STRING(100)   NOT NULL,
    aggregate_id   STRING(100)   NOT NULL,
    event_type     STRING(100)   NOT NULL,
    payload        BYTES(MAX)    NOT NULL,
    occurred_at    TIMESTAMP     NOT NULL,
    created_at     TIMESTAMP     NOT NULL,
    processed_at   TIMESTAMP,
    attempt_count  INT64         NOT NULL DEFAULT (0),
    last_error     STRING(1000),
) PRIMARY KEY (id);
````

**Индексы:**

| Индекс                 | Колонки                                                        | Назначение                                                                                                                                                     |
| ---------------------- | -------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `idx_outbox_pending`   | `NULL_FILTERED on (created_at ASC) WHERE processed_at IS NULL` | Воркер сканирует необработанные записи в FIFO порядке. NULL_FILTERED индекс в Spanner исключает уже обработанные записи, сохраняя индекс компактным и быстрым. |
| `idx_outbox_aggregate` | `(aggregate_type, aggregate_id, occurred_at ASC)`              | Поддержка упорядоченного чтения событий по агрегату и удобство отладки.                                                                                        |

**Как предотвращается повторная обработка:**

* Поле `processed_at` (NULL = pending, не NULL = обработано) управляет фильтрацией записей.
* Консьюмер ведёт собственную дедупликацию по `id` (например, через таблицу `processed_events`).
* Поле `attempt_count` отслеживает количество попыток. Превышение лимита → отправка в dead-letter механизм.

## Q3: Out-of-Order Event Processing

**Что может пойти не так:**

Консьюмер сначала обрабатывает `OrderCancelledEvent` (t=1) и переводит заказ в состояние cancelled. Затем приходит `OrderCompletedEvent` (t=0), и заказ становится completed — перезаписывая корректное состояние.

В итоге система оказывается в неконсистентном состоянии, и downstream процессы (доставка, биллинг) работают с устаревшими данными.

**Как обеспечить порядок:**

1. **Партиционирование по aggregate ID.** Все события одного заказа идут в одну партицию (например, Kafka key = order ID), обеспечивая порядок обработки.

2. **Sequence number на уровне агрегата.** Добавить поле `sequence` в outbox. Консьюмер хранит последний обработанный sequence. Если приходит событие с sequence ≤ текущего — оно игнорируется.

3. **Валидация переходов состояния.** Консьюмер должен проверять допустимость перехода. Если заказ уже cancelled — событие completed должно быть отклонено.

## Q4: 10 Million Unprocessed Events — Recovery Strategy

1. **Диагностика.** Определить причину простоя и убедиться, что можно безопасно переработать backlog.
2. **Батчинг.** Обрабатывать записи партиями (500–1000), отсортированными по `created_at ASC`.
3. **Горизонтальное масштабирование.** Запустить несколько воркеров по разным партициям.
4. **Мониторинг лага.** Отслеживать `created_at` самого старого события.
5. **Dead-letter.** Записи с большим `attempt_count` отправлять на ручной разбор.
6. **Backpressure.** Ограничить нагрузку на downstream системы.
7. **Очистка.** Архивировать или удалять обработанные записи.
8. **Post-mortem.** Добавить мониторинг backlog и health-check’и воркеров.

## Q5: Bug with `time.Now()` in OutboxMuts

```go
CreatedAt: time.Now(),
```

**В чём проблема:**

`time.Now()` в репозитории создаёт timestamp, не связанный с моментом возникновения бизнес-события.

Это приводит к:

* **Drift во времени:** `CreatedAt` отличается от реального времени события (`OccurredAt`).
* **Недетерминированности:** Тесты не могут стабильно проверять timestamp.
* **Некорректной сортировке:** `CreatedAt` отражает момент записи, а не момент события.
* **Проблемам при ретраях:** Повторная запись создаёт новый timestamp.

**Правильный подход:**

Использовать время события из домена:

```go
CreatedAt: event.OccurredAt(),
```

Это:

* фиксирует время на уровне бизнес-логики
* делает систему детерминированной
* упрощает тестирование и reasoning
